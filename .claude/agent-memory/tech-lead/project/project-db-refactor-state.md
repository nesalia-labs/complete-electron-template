As of 2026-06-19, the database and API packages have been refactored to senior patterns:

**packages/db** — `initDatabase()` factory pure (no global `_db`), `closeSqlite()` for cleanup, `runMigrations()` at boot, `drizzle({ client, schema })` for relational types. `AppDatabase` type exported. Migrations live in `packages/db/drizzle/` (auto-generated by `pnpm db:generate`, copied to `dist/drizzle/` at build time).

**packages/api** — `createRouter(db)` factory closes `db` into each procedure's handler. No `os.$context` used (kept things simpler than the DI-via-context approach). `AppRouter = ReturnType<typeof createRouter>`.

**apps/desktop main** — boot order: `initDatabase` → `runMigrations` → `createRouter(handle.db)` → `new RPCHandler(router)`. `app.on('before-quit', () => closeSqlite(handle))` for lifecycle. `handler.upgrade(serverPort)` works WITHOUT a context argument because procedures' context type is `Record<never, never>` (oRPC's `MaybeOptionalOptions` makes it optional).

**Tests** — `packages/db` (4 tests, CRUD + pragmas), `packages/api` (5 tests, full IPC round-trip via Node `MessageChannel`). CI workflows: `test-db.yml`, `test-api.yml`.

**Why:** User explicitly said "le code est à chier, c'est un template mais on a pas les bons patterns" and asked for a senior refactor. Scope validated before coding via AskUserQuestion + EnterPlanMode.

**How to apply:** When the user mentions the db/api packages, these patterns are now established. Don't propose reintroducing singletons, `getDb()`, or untyped query helpers. New tables should follow the same `src/schema/<table>.ts` + factory pattern. See [[feedback-investigate-before-recommending]] for investigation style.
